# ✅ 엔터티 (Entity)
- **정의**: 구별 가능한(Key) 현실 객체의 데이터 표현
<img width="265" height="259" alt="image" src="https://github.com/user-attachments/assets/cfcd71ba-a4b8-4096-89dd-69365cf6285c" />
  
  - **엔터티** : 회원
  - **속성** : 이름, 아이디, 주소, 전화번호
## ■ 강한 엔터티 (Strong Entity)
- 독립적 존재 가능
- 기본키(PK) 포함 → 예: `회원`, `상품`
## ■ 약한 엔터티 (Weak Entity)
- **다른 엔터티에 의존적**
- 자신만의 고유한 PK 없음 → **상위 엔터티의 PK + 자신의 속성 조합**이 PK
- 예: `주문 상세`, `회차별 평가`
# ✅ 릴레이션 (Relation)
<img width="461" height="276" alt="image" src="https://github.com/user-attachments/assets/30fdaafb-8fad-4c97-be64-d3dc8745ab27" />

- **정의**: 엔터티를 표 형식으로 표현한 것 
- 관계형 데이터베이스에선 테이블, NoSQL에선 컬렉션이라고 부른다.
  - RDB → Table (정형 구조, 스키마 고정)
  - NoSQL → Collection (비정형 구조, 스키마 유연)
## ■ 테이블과 컬렉션
- RDB대표 : MySQL → 레코드-테이블-데이터베이스 구조
- NoSQL대표 : mongoDB → 도큐먼트-컬렉션-데이터베이스 구조

###  ▶ ex) RDB
<img width="516" height="282" alt="image" src="https://github.com/user-attachments/assets/b84638d0-2781-4d90-81ca-3f9c06980617" />

- 행(Row) = 레코드(Record) = 튜플: 한 객체(엔터티 인스턴스)의 데이터 집합
- 열(Column) = 필드(Field): 특정 속성의 데이터
- 테이블(Table) : 정보를 행(Row)과 열(Column)로 구성한 구조
- 데이터베이스 : 테이블들의 집합, 테이블 간 관계(Relationship)를 통해 데이터 연결
# ✅ 속성
- **정의**: 엔터티의 특성 정보 → **컬럼(column)** 또는 **필드(field)**
- 예: `회원` 엔터티의 속성 = 이름, 생년월일, 이메일 등
# ✅ 도메인
<img width="282" height="384" alt="image" src="https://github.com/user-attachments/assets/a127ed5f-ec1b-4ab3-969f-848cb3dc93a6" />

- **정의**: 속성이 가질 수 있는 값의 범위
- `성별`의 도메인 = 남, 여
- `생일`의 도메인 = 날짜 형식(YYYY-MM-DD)
# ✅ 필드와 레코드
<img width="450" height="202" alt="image" src="https://github.com/user-attachments/assets/24c6f00e-9221-488f-954e-8eb8d68815b3" />

- 필드(Field) = 속성 하나 → 열
- 레코드(Record) = 속성들의 집합 → 행
## ■ 필드타입
- 필드는 타입을 갖는다. ex) 이름 → 문자열, 전화번호 → 숫자
- 타입은 DBMS마다 다르며, 여기선 MySQL을 기준으로 설명한다
### ▶ 숫자타입
<img width="627" height="249" alt="image" src="https://github.com/user-attachments/assets/76ebb1e6-f4f2-42b7-a2ce-fdc9743f4d2e" />

- `TINYINT` / `SMALLINT` / `MEDIUMINT` / `INT` / `BIGINT`: 크기·범위 다른 정수형
- `DECIMAL(p,s)`: 고정 소수점, 금액·정밀 계산
- `FLOAT` / `DOUBLE`: 부동 소수점, 근사치 계산
### ▶ 날짜타입
- `DATE`: 날짜만 저장 (YYYY-MM-DD)
- `DATETIME`: 날짜 + 시간 (YYYY-MM-DD HH:MM:SS)
- `TIMESTAMP`: UTC 기준 **시점 저장**, 타임존 반영 가능
### ▶ 문자타입
- `CHAR(n)`: **고정 길이**, 성능 빠름, 낭비 있음
- `VARCHAR(n)`: **가변 길이**, 공간 절약
- `TEXT`: 긴 텍스트 (2^16 바이트까지)
- `BLOB`: 바이너리 대용량 객체 (이미지, 동영상 등)
- `ENUM`: 정해진 문자열 집합 중 하나 (e.g. 'male', 'female')
- `SET`: 다중 선택 가능 (e.g. 'music','movie','sports')

# ✅ 키 (Key)
<img width="303" height="286" alt="image" src="https://github.com/user-attachments/assets/a0bcf326-a277-4260-9b7e-2072e0ae2210" />

- 키 : 테이블에서 데이터를 식별(pk) 또는 참조(fk)하는 속성 집합
- 슈퍼키 : 유일성 만족
- 후보키 : 유일성 + 최소성 만족
- 대체키 : 후보키 중 기본키로 선택되지 않은 키
### ▶ 유일성
- 테이블에 있는 모든 행이 **해당 키 값으로 서로 구별 가능해야 함**
- 즉, **중복된 값이 있으면 안 됨**
- 예: 회원 테이블에서 `회원ID`가 1, 2, 3 … 전부 달라야 함
<img width="372" height="214" alt="image" src="https://github.com/user-attachments/assets/ee6b4d4a-1225-45e9-b1d2-9f509fe09a76" />

- 위 그림의 ID는 중복이 있어서 유일성을 만족하지 않는다
### ▶ 최소성
- 유일성을 만족하는 속성 집합에서 **불필요한 속성을 제거한 상태**
- 즉, **꼭 필요한 속성만 포함해야 함**
- 예: (`회원ID`, `이름`)이 유일할때 → `회원ID` 하나로도 유일성이 성립하면 `이름`은 불필요
<img width="271" height="285" alt="image" src="https://github.com/user-attachments/assets/43390d8a-27e0-4854-ab7b-5581af15c534" />

- 위 그림은 {ID, name}이라는 복합키도 유일성이 성립하지만, ID로만 유일성이 성립하기 때문에 {ID, name}은 최소성을 만족하지 않는다
## ■ 기본키 (Primary Key)
- 후보키 중에서 **대표로 선택된 키**
- Null 불가, 중복 불가
### ▶ 자연키 (Natural Key)
- 원래 존재하는 고유 속성
- 예: 주민번호, 학번
### ▶ 인조키 (Surrogate Key)
- 인위적 ID 생성 (AUTO_INCREMENT, UUID)
- 실무에서 선호 (자연키 변경 가능성 방지)
## ■ 외래키 (Foreign Key)
<img width="425" height="429" alt="image" src="https://github.com/user-attachments/assets/e605776c-c085-4c84-83cc-32732d76a8fd" />

- **다른 테이블의 PK 참조**
- 레코드 값 중복 가능
- 관계 표현 + 무결성 보장
- ON DELETE CASCADE 같은 정책 가능
## ■ 후보키 (Candidate Key)
- **유일성 + 최소성** 만족
- 레코드를 유일하게 식별하며, 불필요한 속성이 없음
- 예: 주민번호, 회원ID
## ■ 대체키 (Alternate Key)
- 후보키 중 **기본키로 선택되지 않은 키**
- 예: 주민번호(기본키가 회원ID일 경우)
## ■ 슈퍼키 (Super Key)
- **유일성** 만족
- 최소성 필요 없음 → 불필요한 속성이 포함될 수 있음
- 예: (주민번호), (주민번호+이름), (회원ID+전화번호)
